# BT-AI2c H1 simulation summary

## Section 0. Non-claim disclaimer

This is a 100-seed cycle-level SimPy simulation at exactly one cell
(workload=full, threshold=8). It is not a silicon measurement, not a
multi-cell sweep, and not an end-to-end functional verification of the
proposed speculative-read mechanism. The result below bounds an
optimistic ceiling under the H1 cost model, nothing more. It does NOT
falsify or confirm the BT-AI2c design at silicon scale.

## Section 1. What was modified

The parent simulator at experiments/anomaly/btAI2_honesty_bit_scheduler.py
charges a one-cycle "pipeline bubble" each time the write-barrier refuses
a hypothesis write below threshold (line 571: yield env.timeout(1)). The
H1 hypothesis from btAI2c_design.md sec.2 says that with a speculative-
read path plus rollback amortized to ~0 (assuming low rollback rate),
this bubble can be eliminated.

The H1 wrapper subclasses HonestyBitScheduler and overrides tile_process.
The sole behavioural change is replacing yield env.timeout(1) with
yield env.timeout(0) on the barrier-block branch. All other state
mutations are kept byte-for-byte identical: refused_writes counter,
hyp_writes_blocked counter, effective_prov[tid] poison marking,
produced[tid] timestamp, _enqueue_consumers call, last_activity update.

The original simulator file was NOT touched.

## Section 3. Verdict on F-AI2c-A

F-AI2c-A acceptance criterion: H1-patched mean drop <= 5% AND
seeds_breaching_5pct == 0.

H1 mean drop = 0.0000 <= 0.05  : satisfied.
H1 seeds breaching 5% = 0      : satisfied.

Verdict: PASS.

The decisive observation is not merely that the mean fell below 5%, but
that every single seed shows exactly zero drop. The H1 wrapper does not
spend any extra cycles relative to baseline because, in this model, the
only mechanism by which provenance+barrier degraded throughput was the
+1-cycle bubble; once removed, the simulator runs at baseline
throughput by construction. This is consistent with H1 as stated in
btAI2c_design.md sec.2.

## Section 4. Implication for parent design grade

The parent design BT-AI2c was previously held at design-MEDIUM pending
F-AI2c-A clearance at this cell. With the simulation-level PASS recorded
above, a follow-up amendment cycle MAY consider restoring the parent
design grade to design-HIGH, conditioned on the limitations in
sections 5 and 6 below.

This report does NOT itself amend the parent design file. Any grade
restoration must go through the standard amendment cycle (own#1
honesty-triad, atlas-agent + KG-agent + falsifier-agent triple sign-off)
and must explicitly carry the rollback-rate-zero caveat forward.

## Section 5. Honesty-triad self-evaluation

own#1 (no-fabrication): the four numbers in section 2's H1 column
(mean=std=p50=p99=max=0.0000) and the original column come directly
from reports/anomaly/btAI2c_h1_results.json which was generated by the
in-process sweep runner. No values are hand-edited. The runner uses
the same build_workload, HonestyBitScheduler, and run_simulation entry
points as the parent BT-AI2 simulator; only the barrier-block bubble
was changed.

own#1 (constraint-honesty): the result is OPTIMISTIC. The H1 model
implicitly assumes rollback_rate = 0. Real silicon with non-zero
rollback would emit additional bubbles each time a speculative read
must be reverted; the observed drop on silicon would therefore be
strictly larger than the 0.0000 reported here. A future BT-AI2c
revision should sweep rollback_rate explicitly and re-evaluate
F-AI2c-A at each rate.

own#1 (write-barrier): this report wrote three files only - the
simulator wrapper, the sweep runner, and this summary. No edits to
atlas, KG, domains.json, README.md, inventory.json, or the parent
session file.

## Section 6. Limitations

- Single cell only: workload=full, threshold=8. Behaviour at other
  thresholds (e.g. 6, 7, 9, 10) is not measured here. The robustness
  sweep that motivated this amendment used a wider grid.
- rollback_rate = 0 assumption (see section 5). The H1 result is an
  upper bound on the achievable drop, not a faithful silicon estimate.
- 10000-cycle budget: longer simulations could surface tail effects
  not captured in this run.
- The simulator models the n6 CGRA tile abstraction at cycle level
  but does not include cache hierarchy, off-chip bandwidth, or the
  full atlas-pinned silicon parameter set.
- F-AI2c-B (false-positive promotion-refusal rate) is NOT evaluated
  in this report. It uses the same _maybe_false_positive hook as the
  parent simulator and is unchanged by the H1 patch.

## Section 7. What this report does NOT claim

- It does not claim that BT-AI2c is silicon-ready.
- It does not claim that the speculative-read + rollback mechanism is
  buildable at the cited PROVENANCE_BIT_OVERHEAD = phi/sigma_n = 1/36.
- It does not claim that rollback_rate is in fact low on real
  workloads; that is a hypothesis the next BT cycle must test.
- It does not claim that F-AI2c-A passing at this single cell implies
  passing across the broader (workload, threshold) grid.
- It does not amend the parent design grade. Any grade change is the
  responsibility of a separate amendment cycle.
- It does not modify atlas, KG, domains.json, inventory, or any
  README.

End of summary.
