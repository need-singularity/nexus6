## 0. Scope recap

Phase 1 decision (2026-04-23): keep `sigma=12, tau=4, phi=2` inlined in all 18
`verify_*.hexa` files because the hexa stage does not support stable cross-unit
`import`/`include` of typed constants. SSOT is `atlas/n6_core_constants.hexa`,
reviewer uses it as the diff target.

Phase 2 question: can we unify the 18 files now, and if not fully, how close
can we get without waiting for the lang-side fix?

## 1. Finding — include mechanism DOES already exist in `$HEXA_LANG`

Investigation of `~/core/hexa-lang/self/` on 2026-04-24 shows:

| Artifact | Kind | Status |
| -------- | ---- | ------ |
| `self/module_loader.hexa` (474 lines, Phase2 C1, rewritten 2026-04-18) | Iterative-DFS **text preprocessor** that expands `use "path"` / `import "path"` directives leaves-first, strips the directives, and drops top-level `pub ` prefixes. | Working; used by bootstrap. |
| `tool/flatten_imports.hexa` | Sibling preprocessor, same contract, used by the C-codegen path (which also treats `use` as a comment). | Working; called explicitly from build scripts. |
| `self/test_module_import.hexa`, `self/test_deep_import_chain.hexa` | Regression tests — deep chain (ROI #149) was the silent-exit bug; fixed. | Green. |
| `self/modules/` (boot_lexer, boot_parser, boot_compiler, boot_structs, boot_typecheck) | Live callers of the directive form. | In production. |

**What this means for canon:**

- The "stable `import` in the hexa compiler itself" is still not there — the
  compiler/interpreter proper **does not resolve `use` directives at run time**.
  They are treated as comments unless a preprocessor expands them first.
- But there IS a **documented, stable text-inline preprocessor** (`module_loader` /
  `flatten_imports`) that is already used inside `hexa-lang` for exactly this
  shape of problem (N modules wanting to share a constants file, no runtime
  resolution). It expands `use "path"` into the textual body of the referenced
  `.hexa` file, recursively, in topo order.
- The "gap" `hxa-20260422-00?` is narrower than originally written: it is about
  **compiler-native, typed `use`** — not about *any* module inlining. A CI
  preflight that invokes the preprocessor closes the operational gap today.

## 2. Staged plan

### Stage A (interim, **unblocked today**) — CI-generated `_shared_constants.hexa`

Goal: get the 18 files down to a single textual copy of the constants, with no
hand-edit divergence risk, without waiting for lang landing.

**Design.**

1. Canonical SSOT stays at `atlas/n6_core_constants.hexa` (already exists).
   Rewrite it to expose **bare top-level `let`s inside a designated `fn
   n6_core_constants_inject()` body**, so the expansion site is textually
   explicit. (Top-level `let` across files is itself a lang gap; wrapping in a
   helper fn keeps the stage0 parser happy.)

2. Add a new generator script: `scripts/gen_shared_constants.sh` (bash, no hexa
   dependency — intentional, so pre-commit works on a clean checkout). It emits
   `canonshared/_shared_constants.hexa` by reading the SSOT and stripping the
   function wrapper, leaving just the six `let` decls + identity `assert`s, with
   a header banner:

   ```
   // AUTO-GENERATED by scripts/gen_shared_constants.sh from
   // atlas/n6_core_constants.hexa — DO NOT EDIT BY HAND.
   ```

3. Each of the 18 `verify_*.hexa` files replaces its six `let` lines with a
   single directive:

   ```hexa
   use "canonshared/_shared_constants"
   ```

   Comment-style note: `use "..."` is parsed by `module_loader` regardless of
   whether the host file uses `//` or `#` comments, because the matcher keys on
   the directive literal at the start of a trimmed line. (Confirmed by reading
   `ml_strip_and_clean` in `self/module_loader.hexa`:265+.)

4. Runner contract for "run a single verify file in isolation":

   ```
   hexa tool/flatten_imports.hexa domains/.../verify_X.hexa /tmp/verify_X.flat.hexa
   hexa /tmp/verify_X.flat.hexa
   ```

   Wrap this in `bin/n6_verify_run` (2-line shim) so authors don't type the
   two-step. Preserves the Phase-1 self-containment invariant *in spirit* —
   any human or CI job can reproduce the run from the verify file alone,
   because flatten_imports is deterministic and has no network/side-effects.

5. CI preflight. Add a job to `.github/workflows/ci.yml` that:
   - regenerates `canonshared/_shared_constants.hexa` from the SSOT;
   - fails if the regenerated content differs from the committed content
     (classic "generator clean" check — same pattern as
     `cdo-validate` already does for JSON);
   - runs `tool/flatten_imports` on each of the 18 files and asserts the
     resulting binary still passes (identity holds, exit 0).

6. Pre-commit hook extension. Mirror the `pre-commit-n6-integrity` shape
   (`hooks/pre-commit-n6-integrity`): if any staged file is under one of the
   18 domains AND touches the sigma/tau/phi literals, block the commit with
   a pointer to the SSOT. (Drift detection, not generation — generation lives
   in CI so Windows contributors aren't blocked on a local bash prereq.)

**What Stage A buys.**

- Drift surface shrinks from **18 text copies → 1**
  (`atlas/n6_core_constants.hexa`).
- The six `let` values live in exactly one human-edited file. The generated
  `_shared_constants.hexa` is a CI artifact, not a source of truth.
- Phase-1's self-containment property is preserved *up to a documented
  preflight step* — `flatten_imports` is part of the standard hexa toolchain,
  not a bespoke n6 script.
- Zero dependency on any new hexa-lang feature. Ships this week.

**What Stage A does NOT buy.**

- `hexa run verify_X.hexa` directly (without flatten) still fails — the `use`
  directive is a comment to the interpreter until the preprocessor fires.
  This is the residual value in waiting for Stage B.
- Typed constant exposure — everything is still `let x = 12`-style, not
  `pub const SIGMA: i64 = 12`. Type-safety parity with rust/ts sibling
  modules is deferred to Stage B.

### Stage B (long-term, lang-side) — real `import`/`use`

Goal: eliminate the preflight step entirely. `hexa run verify_X.hexa` resolves
`use "atlas/n6_core_constants"` natively; constants are typed at the module
boundary; a downstream edit to the SSOT propagates at compile time without a
codegen step.

**Spec, in the shape the hexa-lang inbox expects** (`bin/proposal_inbox
submit --to hexa-lang --category lang_gap --kind cluster ...`):

1. **Syntax.** Keep `use "relative/path"` (string-literal form) as the one
   surface; the preprocessor form becomes a semantically equivalent fallback.
   No new keyword; no path-aliased form in v1 (defer `use atlas::constants::*`).

2. **Typed export surface.** Add `pub const NAME: T = expr` at module top level.
   v1 restriction: `expr` must be a constant expression over
   `i64|u64|f64|bool|string` only — same subset the comptime evaluator already
   handles for `@pure` fns. Recursion and fn calls disallowed in v1.

3. **Resolution order.** `use "x"` searches, in order:
   (a) path relative to importer file's directory;
   (b) `HEXA_LIB_PATH` entries (already exists as env in `self/`);
   (c) project root (`.hexa-init` marker).
   Same rules `module_loader` already implements — lifts them into the
   compiler proper.

4. **Cycle + depth.** Reuse the iterative-DFS + 64-bucket hashset that
   `self/module_loader.hexa` already validated (the ROI #149 deep-chain fix is
   a reference implementation). Error on cycle with the detected back-edge
   printed; hard cap at 64 transitive imports per translation unit (empirically
   enough — `native_compile_poc.hexa` peaks at 41).

5. **Compat contract.** For one release after Stage B lands, files that
   contain `use "..."` continue to work under `flatten_imports` preflight
   (backward compat). A `--strict-imports` interpreter flag opts into "fail
   if unresolved `use` is encountered", and CI flips to it in the release
   *after*. Two-release migration window.

6. **Acceptance tests.** Port `self/test_module_import.hexa` and
   `self/test_deep_import_chain.hexa` from preprocessor harness to compiler
   harness. Add an n6-side test that imports
   `atlas/n6_core_constants` and asserts `SIGMA * PHI == N * TAU`.

**Forwarding.** Per `config/lang_gap_forward_convention.json`, file this as a
`cluster` kind lang_gap through `hexa ~/core/hexa-lang/bin/proposal_inbox
submit --to hexa-lang --category lang_gap --kind cluster --title 'typed const
import across translation units' --from canon`. Attach this proposal
and the upstream `module_loader.hexa` reference so the reviewer sees the
implementation already exists as a preprocessor.

### Stage C (cleanup, post-Stage-B) — drop the preflight

One PR per domain-family (cognitive / compute / physics / energy / infra) that:

- removes `scripts/gen_shared_constants.sh`;
- removes `canonshared/_shared_constants.hexa`;
- rewrites the `verify_*.hexa` head to the typed form
  `use "atlas/n6_core_constants"; use atlas::n6_core_constants::{SIGMA, TAU,
  PHI, N, SOPFR, J2};`
- deletes the CI preflight job;
- keeps the identity assert (the one piece of domain-local proof that
  survives regardless of how constants enter the file).

Five PRs, ~4 files each, fully mechanical. Gated on Stage B green + one
release in `--strict-imports` mode.

## 4. Risks & tradeoffs

| Risk | Mitigation |
| ---- | ---------- |
| `flatten_imports` is a `hexa-lang` tool; n6 taking a hard dep on it widens the coupling surface. | Vendor `flatten_imports` as a path reference in `config/engine_manifest.json` (already has `hexa_lang_path` field). Pin a commit SHA. If upstream removes it, n6 falls back to a 50-line python reimpl — the format is trivial. |
| CI-generated file drift (dev forgets to regenerate). | Generator-clean check in CI blocks merge. Pattern already in use for `n6_manifest.json`. |
| Phase-1's "each file runs standalone in `hexa run`" invariant is weakened. | Ship `bin/n6_verify_run` wrapper as the canonical runner. Document in each verify file's header (one line). Self-containment shifts from "one file" to "one file + one documented tool" — acceptable given the drift reduction. |
| Stage-B spec is rejected or deprioritized upstream. | Stage A is complete value on its own. Worst case we stay on preflight forever — still 18× drift reduction vs Phase 1. |
| Verify files use `#` comments, not `//`. | `module_loader.ml_strip_and_clean` matches on the directive literal, not on comment tokens (verified 2026-04-24). `use "..."` works regardless of the host file's comment style. |
